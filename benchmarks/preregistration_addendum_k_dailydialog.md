## Motivation

Study S1 (LoCoMo) tested AFT on a factual open-domain QA benchmark and produced a clear
negative result (H1/H2 FAIL, Δ < 0). This result is consistent with the theoretical scope
of AFT: the library is designed for *affective* memory retrieval, not for general factual
recall. A naive cosine-based RAG has no disadvantage in factual recall regimes because the
relevant signal is lexical/semantic overlap, which cosine similarity handles well.

The open question is: **does the AFT advantage observed in `realistic_recall_v2` generalise
beyond the curated controlled-replay benchmark?** Specifically, does AFT outperform naive
cosine when the query *requires affect-awareness* to select the correct memory — in dialogue
content generated by real speakers (not by the author)?

This addendum registers that question as Hypothesis Hk1 and pre-declares the statistical
decision rule, dataset construction protocol, and downstream paper changes **before** any
data are loaded, personas are built, or benchmarks are run.

---

## Dataset rationale — DailyDialog

DailyDialog (Li et al. 2017) is selected over LOCOMO, NarrativeQA, MELD, and MSC for the
following reasons:

| Criterion | DailyDialog | Rejected alternatives |
|---|---|---|
| Affect-laden content | ✓ Turn-level Ekman 7-class emotion labels | LOCOMO: factual QA (known FAIL regime); NarrativeQA: factual; LongMemEval: factual |
| Maps to PAD space | ✓ Ekman → PAD canonical mapping (same as `appraisal_keyword.py`) | MELD: TV-scripted (not ecological everyday); MSC: no explicit emotion labels |
| Ecological everyday content | ✓ 13,118 dialogues: work, school, travel, relationships, health | MELD: Friends TV show (scripted) |
| Public + reproducible | ✓ HF Hub `daily_dialog`, CC BY-NC-SA 4.0 | — |
| Citation depth | ✓ 2000+ citations (COLING 2017) | — |

**Accepted trade-off:** DailyDialog dialogues are short (~8 turns, single-session).
Multi-session memory structure is introduced by concatenating 4–5 dialogues per synthetic
persona (see §Dataset construction below). The spoken content remains real and unaltered;
only the session-boundary structure is synthesised. This is disclosed explicitly as
"ecologically-grounded synthesis" in the pre-registration, paper, and closure.

## Hypothesis Hk1

> On `benchmarks/datasets/dailydialog_personas_v1.json` — N=120 synthetic personas,
> 4 queries per persona (480 queries total), embedder `multilingual-e5-small`, top_k=2
> — AFT `top1_accuracy` > naive_cosine `top1_accuracy` on aggregate affect-conditioned
> recall AND on ≥ 2 of 3 query types (see §Query types below).

One-tailed (directional): the sign of Δ is expected positive based on the affect-aware
retrieval theory and the realistic_recall_v2 results.

## Dataset construction protocol

This section is committed **before** generating `dailydialog_personas_v1.json` to prevent
data-snooping.

### Persona builder algorithm

1. Load all DailyDialog dialogues from HF Hub (`daily_dialog`, `train` + `validation` splits).
2. Compute per-dialogue emotion density: fraction of turns with label ≠ `no_emotion` (label 0).
3. **Emotion-density filter:** retain only dialogues with emotion density ≥ 0.30 (≥ 30% of turns carry an explicit emotion label). This eliminates the ~83% majority-null dialogues while preserving natural conversational context.
4. **Topic balance check:** ensure retained pool spans all 10 DailyDialog topic categories.
5. For each of 120 personas (seed=0 for reproducibility):
   a. Sample 4–5 dialogues uniformly at random from the filtered pool **without replacement within a persona** (across-persona repetition is acceptable for pool size).
   b. Each sampled dialogue becomes one session (`session_id = f"session_{i}"`).
   c. Session turns are kept verbatim; emotion labels and topic indices are carried through.
6. For each persona, compute per-session PAD vectors using the canonical Ekman→PAD mapping:

| Ekman label | valence | arousal | dominance |
|---|---|---|---|
| `happy` (1) | +0.70 | +0.50 | +0.60 |
| `surprise` (2) | +0.05 | +0.80 | −0.20 |
| `sadness` (3) | −0.60 | +0.20 | −0.40 |
| `anger` (4) | −0.50 | +0.75 | +0.60 |
| `disgust` (5) | −0.60 | +0.40 | +0.30 |
| `fear` (6) | −0.55 | +0.75 | −0.60 |
| `no_emotion` (0) | 0.00 | 0.20 | 0.50 |

Per-session affect state: **majority-vote emotion** of turns in that session → PAD vector.
(Sessions with majority `no_emotion` are allowed but are less likely post-filter.)

### Query types

4 queries per persona; 1 per type for the first 3 types, then 1 "trajectory" control query:

**Type 1 — Emotion-state recall:**
"In session {N}, what emotion was expressed when discussing {topic}?"
- *Ground-truth session:* the session where topic matches and the majority emotion label is
  the queried emotion.
- *Distractor pool:* other sessions of the same persona with the same topic but different
  dominant emotion. Requires ≥ 1 valid distractor; persona is regenerated if not satisfied.

**Type 2 — Affect-conditioned content:**
"When the user felt {emotion}, what was the main topic of the conversation?"
- *Ground-truth session:* session with majority emotion = the queried emotion.
- *Distractor pool:* sessions with different dominant emotion but same topic index.

**Type 3 — Affective trajectory:**
"Did the conversation about {topic} leave the user feeling better or worse emotionally?"
- *Ground truth:* session where valence-of-last-turn > valence-of-first-turn (positive shift)
  or vice versa (negative shift); answer label is "positive" or "negative".
- This type tests whether AFT's momentum and decay signals — which track temporal valence
  change — give it an edge over static cosine similarity.

**Type 4 — Same-topic cross-session control (distractor hardening):**
"Of the conversations about {topic}, which one happened when the user was least stressed?"
- Ground truth: session with lowest arousal score (from PAD majority-vote).
- All sessions discussing the same topic are distractors; top_k=2 must select the correct one.

The query text is generated **programmatically from templates** using only the emotion label,
topic index, and valence/arousal values already present in the persona data. **No LLM is
in-loop for ground-truth assignment.** This guarantees determinism and prevents LLM-aware
bias in the evaluation signal.

## Statistical analysis plan (pre-declared)

- **Comparison:** AFT system vs naive_cosine baseline (same as all previous closures).
- **Primary metric:** `top1_accuracy` (top-1 hit among top_k=2 returned memories).
- **Secondary metric:** `hit_at_k` (any correct memory in top-2).
- **Test:** Paired bootstrap difference, n=10,000, seed=0, one-tailed.
- **McNemar test:** exact two-tailed for binary correctness vectors.
- **Effect size:** Cohen's d (paired differences normalised by SD of differences).
- **Family correction:** Holm–Bonferroni, m=4 (aggregate + 3 directional query types;
  Type 4 control is reported but not in the Holm family).
- **N:** 120 personas × 4 queries = 480 total queries.
- **Bootstrap CI:** 95% percentile CI on Δ.

---

## Decision rule (pre-declared, ex-ante)

A language/system pair **passes Hk1** if and only if:

1. `p_bootstrap` < 0.05 (one-tailed) on aggregate `top1_accuracy`, after Holm correction.
2. Δ (`top1_accuracy` AFT − naive_cosine) > 0.
3. 95% bootstrap CI does not cross 0 (all-positive).
4. ≥ 2 of 3 directional query types (Type 1, Type 2, Type 3) show Δ > 0 individually.

**Marginal handling:** if `0.04 < p_holm < 0.05`, classify as "PASS marginal" and flag in
the closure. No post-hoc adjustment to effect size thresholds.

### Branch A — PASS

AFT `top1_accuracy` > naive_cosine, p_holm < 0.05, CI all-positive, ≥ 2/3 types Δ > 0.

Downstream consequences:
- Paper §6 adds a "Cross-domain replication" subsection citing Hk1 numbers.
- §8 limitations reduces "controlled-replay only" caveat.
- `claim_validation_matrix.json`: `cross_domain_affect_replication` status → `established`.
- Abstract may cite "extended to ecologically-grounded dialogue synthesis (DailyDialog)".

### Branch B — FAIL

Any of the pass conditions unmet.

Downstream consequences:
- Paper §8 explicitly states: "AFT's advantage over naive cosine is observed within curated
  controlled-replay benchmarks; a pre-registered test on affect-conditioned DailyDialog
  dialogues (Hk1) did not replicate the effect (Δ={observed}, p_holm={observed})."
- §6 does not claim cross-domain replication.
- `claim_validation_matrix.json`: `cross_domain_affect_replication` status → `not_established`.
- Abstract does not add DailyDialog claim.
- This is a publishable scientific result: it scopes AFT to the curated affect-aware regime.

## Coherence with prior closures

| Study | Dataset | Hypothesis | Regime | Result |
|---|---|---|---|---|
| S1 (LoCoMo) | External open-domain QA | H1 (F1), H2 (judge-acc) | Factual QA | **FAIL** (Δ < 0) |
| Hd2 EN N=200 (realistic_recall_v2) | Curated replay benchmark | AFT > naive_cosine | Affect-aware curated | **PASS** |
| Hd2_IT/ES.me5 N=120 (Branch C) | Curated replay IT/ES | AFT > naive_cosine | Affect-aware curated (multilingual) | **FAIL** |
| **Hk1 (this addendum)** | **DailyDialog (real dialogue + synthetic personas)** | **AFT > naive_cosine** | **Affect-conditioned ecological** | *pending* |

Hk1 fills the **ecological affect-conditioned** cell, which is orthogonal to all prior
studies. A PASS result would establish that AFT's advantage is not confined to the
`realistic_recall_v2` dataset, while acknowledging the LoCoMo FAIL regime (factual).
A FAIL result confirms that the advantage is dataset-specific in the curated regime.

**Neither outcome invalidates the headline EN result.** Both are pre-declared and publishable.
