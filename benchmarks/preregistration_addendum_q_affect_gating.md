## Motivation

Addendum P closed the affect-free architecture-vs-cosine line with a sharp negative: with the
recalibrated M1 mapping, always-on dual-path LLM appraisal **significantly underperforms** naive
cosine on affect-free queries (top1 0.800 vs 0.887, Δ=−0.0875, p=0.0018). Yet the same study
showed the affect channel is **not noise**: dual > neutral (Hp2 PASS) and dual > sync (Hp3 PASS,
d=0.95). Combined with the oracle-affect positives (Hd1, Hd2, Addendum M FR), the evidence
triangulates to one untested synthesis, named in the Addendum P closure as the next angle:

> apply affect only when the query is affect-discriminative — rather than always blending it.

This is **gating** (on/off), not the per-type _weight_ routing of Addendum L (which failed on
LoCoMo, where no AFT configuration beat naive RAG on any category). Three questions, one study:

1. **Hq1 — the missing quadrant.** Oracle-affect AFT wins on affect-discriminative queries
   (Hd1/Hd2); LLM-inferred affect loses on affect-free queries (Hp1). Does **LLM-inferred**
   affect win on **affect-discriminative** queries? Never tested.
2. **Hq2 — does gating recover the affect-free penalty** relative to always-on blending?
3. **Hq3 — headline.** Does gated AFT beat naive cosine on a **mixed** workload where neither
   pure strategy can win everywhere?

## Mechanistic premise (pre-stated, constrains dataset design)

AFT retrieval signals are **state-based**: mood congruence, affect proximity and momentum
compare the _runtime affective state_ with each memory's tag — the query text itself is never
appraised. Therefore an "affect-discriminative" query in a noAF design must be one where:

- the target event and a semantic distractor are near-ties on embedding similarity, and
- the target's (LLM-inferred) affect is **congruent with the end-of-arc runtime state** while
  the distractor's is not, so the mood-congruence/affect-proximity signals break the tie.

This is the honest operationalization available to the current architecture. If Hq1 fails, the
pre-specified reading (Branch C) is that state-based affect signals are insufficient for
query-driven affect discrimination, and the next mechanism is retrieve-time **query appraisal**
(a new signal), not routing.

## Dataset (pre-specified): `realistic_recall_v5_gate`

Generated by a scripted generator `benchmarks/datasets/generate_v5_gate.py` (gpt-5-mini, one
structured-JSON scenario per call), mirroring the Addendum P method:

- **Scenario IDs disjoint from v3 and v4**: new namespace `q01_`–`q50_` (none of `s01..s125`,
  none of `p01..p45`). Themes drawn from a fresh 50-item list, thematically disjoint from both
  prior theme lists.
- **50 scenarios target (≥48 floor), 4 queries each → 200 queries (≥192 floor)**, exactly
  **2 affect-free + 2 affective per scenario** (≥96 per half).
- Skeleton identical to v4: 6 events (session 1 = e1–e3, session 2 = e4–e6), two-session
  emotional arc, all queries attached to session 2 (asked after the full arc is encoded).
- **No `valence`/`arousal` fields** on any event; **no `state` field** on queries. The LLM
  appraisal engine is the only affect source at encode time.
- Every query carries a **`gate_label`** field: `"affect_free"` or `"affective"` — the ground
  truth for the oracle gate and for gate-classifier accuracy reporting.
- **Affect-free half** (gate_label=affect_free): challenge types rotated from
  {`semantic_confound`, `recency_confound`, `same_topic_distractor`} (2 of 3 per scenario) —
  semantically determinable, the regime where cosine won in Addendum P.
- **Affective half** (gate_label=affective), one of each per scenario:
    - `affect_congruent_tiebreak`: the query is a near-paraphrase tie between the target and a
      distractor event; the target's emotional tone matches the **end-of-arc** tone, the
      distractor's is opposite. Events must convey emotion through behaviour/imagery **without
      naming the emotion**, and the query must not reuse the emotion-bearing words of either
      event (minimizing lexical leakage that would make cosine win for free).
    - `affective_arc_blind`: the query references the emotional trajectory only (e.g. "the moment
      everything finally turned"), targeting the turning-point event, with minimal content-word
      overlap with the target text.
- **Author-blind**: the generator never runs any retrieval system; the dataset JSON is
  validated (schema + structural constraints + gate-label balance) and **committed before any
  benchmark run**.
- Reproducibility: the committed JSON is the canonical frozen artifact; `default_top_k = 2`
  (as v4).

### Dataset acceptance gates (run before freezing, no LLM systems involved)

1. Schema validation (extended noAF schema with `gate_label`), structural constraints from
   `_validate_dataset` (no dangling ids, non-trivial candidate windows, challenge-type
   inventory), and exact 2/2 gate-label balance per scenario.
2. **Lexical-leakage gate (affective half):** for each `affect_congruent_tiebreak` query, the
   token-overlap (content words) between query and target event must not exceed the overlap
   between query and its distractor event by more than 2 tokens — checked mechanically by the
   generator; scenarios failing it are regenerated. This is a _construction_ constraint, fixed
   before any system runs.

## Systems under test

| System             | Appraisal               | dual_path | Retrieval weights                             |
| ------------------ | ----------------------- | --------- | --------------------------------------------- |
| `naive_cosine`     | none                    | n/a       | pure cosine (control)                         |
| `aft_llm_dual`     | LLMAppraisalEngine (M1) | yes       | base `[0.35,0.25,0.15,0.10,0.10,0.05]` always |
| `aft_gated_oracle` | LLMAppraisalEngine (M1) | yes       | gated by **ground-truth** `gate_label`        |
| `aft_gated_llm`    | LLMAppraisalEngine (M1) | yes       | gated by **LLM gate classifier**              |

Gating uses the existing public routing API (`QueryClassifierConfig(mode="llm",
routed_weights=…)`, engine `query_classifier=`):

- gate says `affect_free` → weights `[1.0, 0, 0, 0, 0, 0]` (pure semantic; by construction the
  ranking coincides with `naive_cosine` on the same embedder — asserted in the dry run),
- gate says `affective` → base weights `[0.35, 0.25, 0.15, 0.10, 0.10, 0.05]` (identical to
  `aft_llm_dual`).

**LLM gate classifier** (frozen with this pre-reg, implemented in the runner): binary prompt —
"does answering this query require tracking emotions/feelings/emotional trajectory
(`affective`), or is it a factual/semantic lookup (`affect_free`)?" — JSON-schema constrained,
SHA-256 LRU cached, fallback label on error = `affective` (preserves full-AFT behaviour).
Per the Hl3 lesson (Addendum L closure post-closure addendum), the study **must report gate
classifier accuracy and the confusion matrix** against `gate_label`, and includes the oracle
arm to bound classifier-induced loss.

**Shared appraisal engine (pre-specified design choice):** the three AFT systems share one
`LLMAppraisalEngine` instance (LRU cache ≥ 1024). Appraisal is a pure function of event
content, so sharing guarantees the three AFT arms see **identical inferred affect**, isolating
the gating variable and removing appraisal sampling noise from the system contrasts. Encode
discipline is identical to Addendum P (`encode` → immediate `elaborate`, dual-path).

Embedder: `sbert-bge` (paper-canonical). Engines otherwise default-configured.

## Hypotheses

### Confirmatory family (Holm-Bonferroni, m=3, α=0.05, one-tailed)

| ID      | Population               | H1                                       | Effect threshold |
| ------- | ------------------------ | ---------------------------------------- | ---------------- |
| **Hq1** | affective subset (N≈100) | `aft_llm_dual.top1 > naive_cosine.top1`  | Δ > 0.05         |
| **Hq2** | full corpus (N≈200)      | `aft_gated_llm.top1 > aft_llm_dual.top1` | Δ > 0.05         |
| **Hq3** | full corpus (N≈200)      | `aft_gated_llm.top1 > naive_cosine.top1` | Δ > 0.05         |

- **Metric:** top1_accuracy (primary, as all prior addenda)
- **Test:** paired bootstrap over per-query correctness, n=10,000 resamples, **seed=0** (frozen)
- **PASS rule per hypothesis:** Δ > 0.05 AND Holm-adjusted one-sided p < 0.05
- Hq1 is evaluated on `aft_llm_dual` (not the gated arms) so it measures the affect channel
  itself, uncontaminated by gate errors; on the affective subset the gated arms are intended
  to coincide with `aft_llm_dual` modulo gate mistakes.

### Exploratory (CIs reported, no pass/fail gate)

- **Hq4:** `aft_gated_oracle` vs `naive_cosine`, full corpus — gating upper bound.
- **Hq5:** `aft_gated_llm` vs `naive_cosine`, affect-free subset — cost of gate
  false-positives (expected ≈ 0 by construction with a perfect gate).
- **Hq6:** gate classifier accuracy, per-label recall, confusion matrix (mandatory report).
- Per-challenge-type top1 decomposition for all systems (descriptive).

## Decision rule / pre-specified branches

- **Branch A (Hq1 PASS ∧ Hq3 PASS):** affect-aware gating yields a deployable advantage on
  mixed workloads. Claim `appraisal_llm_real_dual_path` is **revised, not broadened**: the
  positive claim attaches to the _gated_ configuration; always-on blending remains falsified
  for affect-free regimes. Hq2 then qualifies how much of the gain is gating vs channel.
- **Branch B (Hq1 PASS ∧ Hq3 FAIL):** the affect channel wins where affect discriminates, but
  gate errors and/or the affect-free tie erase the aggregate advantage. Compare Hq4 (oracle) to
  locate the loss: if oracle PASSes where gated-llm fails, the bottleneck is the classifier
  (improvable); if oracle also fails, the affective-half advantage is too small to carry the
  aggregate.
- **Branch C (Hq1 FAIL):** LLM-inferred affect does not beat cosine even on
  affect-discriminative queries under state-based signals. The affect-routing line **closes**
  (no further gating studies on this mechanism); the residual hypothesis space moves to
  retrieve-time query appraisal (new signal, separate pre-registration if pursued). Honest
  negative recorded on the claim matrix.
- In every branch, Hq6 (gate accuracy) and the oracle bound (Hq4) are reported.

## Statistical analysis plan

Identical machinery to Addenda D/G/P (`benchmarks/common/statistics.py`):

- Per-query correctness vectors per system; subsets selected by `gate_label`.
- Paired bootstrap (same query resampled across systems), n=10,000, seed=0.
- Report per hypothesis: Δ, 95% percentile CI, one-sided p (raw and Holm-adjusted within the
  m=3 confirmatory family), Cohen's d (paired).
- No interim looks; single run at the pre-registered N; results reported regardless of outcome.

## Amendment 1 (pre-execution, 2026-06-11) — gating implemented as a front-router

**Discovery (before any dataset generation or benchmark run):** the engine-level routing path
(`QueryClassifierConfig.routed_weights`) feeds the routed vector through `adaptive_weights()`,
which modulates weights by the *current mood* (`retrieval.py`). With a non-neutral mood — the
normal condition mid-arc — a routed `[1,0,0,0,0,0]` acquires non-zero mood-congruence /
affect-proximity components. The gate-off arm would therefore NOT coincide with `naive_cosine`,
breaking the pre-registered dry-run assert and contaminating the Hq2/Hq3 contrasts.

**Revised implementation (replaces the "existing public routing API" paragraph):** the gated
adapters implement gating as a **front-router at the adapter level**:

- every event is encoded into the AFT engine exactly as in `aft_llm_dual` (and, in parallel,
  into a local cosine index identical to `naive_cosine`'s — same embedder instance, same
  content strings);
- at retrieve time the gate classifies the query: `affect_free` → answer from the local cosine
  index (code path shared with `naive_cosine`, equivalence by construction); `affective` →
  `engine.retrieve` with configuration **identical** to `aft_llm_dual`.

**Disclosed consequence:** in the gated arms, affect-free retrievals bypass the engine and so
skip its retrieval side effects (reconsolidation, Hebbian strengthening); `aft_llm_dual`
performs them on every query. This is an intrinsic property of the gated architecture (fewer
state mutations), not a confound introduced by measurement.

Hypotheses, dataset, statistics, seeds and decision rules are **unchanged**. This amendment is
committed before `generate_v5_gate.py` exists and before any data are generated.
